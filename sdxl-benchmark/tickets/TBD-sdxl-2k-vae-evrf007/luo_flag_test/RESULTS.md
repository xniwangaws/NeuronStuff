# kaena-30652: Validation of Luo's `--tiled-inst-limit` flag

**Test environment**: trn2.3xlarge spot in sa-east-1b (i-0a12c61d319cff410), AMI Neuron PT 2.9 Ubuntu 24.04 20260522, SDK 2.29 (compiler 2.25.3371.0+f524f7f8). Ticket was filed against SDK 2.32 but the failure is at the HLO verifier / hlo2tensorizer boundary, which is unchanged at this patch level.

## Verdict
**Partial success**: Luo's flag (`--internal-hlo2tensorizer-options=--tiled-inst-limit 10000000`) **does** bypass NCC_EVRF007. The compile now progresses past the HLO verifier into hlo2penguin/Tensorizer. However, it then fails downstream with **NCC_EXSP001** — a legitimate HBM exhaustion (38.14 GB needed vs 24 GB available on trn2). The arbitrary-limit issue is fixed; what remains is a real memory budget problem.

## Control (existing flags)
```
--tensorizer-options=--inst-count-limit=15000000
--internal-max-instruction-limit=15000000
```
- Compile time: **15.0 s**, exit 70.
- Failure: `[NCC_EVRF007] Instructions generated by compiler 7,691,582 exceeds the typical limit of 5,000,000`.
- Stage: `HLOToTensorizer.runHlo2Tensorizer` (HLO verifier).
- This matches the original ticket fingerprint exactly (same 7,691,582 instruction count).

## Treatment (Luo's flag)
```
--internal-hlo2tensorizer-options=--tiled-inst-limit 10000000
```
Flag syntax: space form (`--tiled-inst-limit 10000000`) accepted; equals form not tested but the space form is what neuronx-cc forwards verbatim to `hlo2penguin` (visible in command line: `... --tiled-inst-limit 10000000 --expand-batch-norm-training ...`).

- Compile time: **53.5 s**, exit 70.
- NCC_EVRF007 **no longer triggers**.
- New failure: `[NCC_EXSP001] Size of HBM memory required for the model exceeds HBM limit of gen3. Needed more than 40,947,923,456 (38.14GB) vs. available bytes 25,769,803,776 (24.00GB)`.
- Stage: `TongaBufferUsageAnalysis.profileMemoryPressure` (much later — buffer-usage analysis on the partitioned graph).
- Internal assert: `Assertion failed: usage <= hbm_size` on operator `_add.2010`.

## Interpretation
Luo's flag changes the failure mode from an arbitrary HLO-instruction-count ceiling to a real, physical HBM constraint. The patch is doing what it claims (raises the tiled-instruction threshold inside hlo2tensorizer so the partitioner can actually emit code for this graph). The remaining NCC_EXSP001 is a separate issue: the SDXL VAE 2K decoder genuinely needs more HBM than a single trn2 chip provides. Reasonable next steps for the customer would be tensor parallelism, tiled VAE decode (e.g. diffusers' `vae.enable_tiling()`), or a multi-chip target.

## Recommendation
- The compiler-team patch is **validated**. Luo's flag should be promoted from a workaround to either (a) the new default tiled-inst-limit, or (b) a documented escape hatch.
- Ticket can move to **Pending Compiler Decision** on whether to default the new limit, OR closed-as-fixed with a note that customers hitting NCC_EXSP001 next need a memory-budget conversation, not an instruction-count one.
- The ticket's original framing ("compile fails despite 15M --inst-count-limit") is resolved: that flag was the wrong knob; `--internal-hlo2tensorizer-options=--tiled-inst-limit ...` is the right one.
