# Compiler Ticket: SDXL VAE decoder NEFF compile fails NCC_EVRF007 at 2048×2048; --inst-count-limit=15M does not lift the limit

## Ticket ID
TBD

## Date
2026-05-25

## Summary

When tracing the SDXL VAE decoder for a **2048×2048 output** image (input latent shape `[1, 4, 256, 256]`, bf16) on trn2.3xlarge, `neuronx-cc` aborts with:

```
[NCC_EVRF007] Instructions generated by compiler 7,691,582 exceeds the typical
limit of 5,000,000. Input computation graph is too big or has large operators.
```

The standard workaround `--tensorizer-options=--inst-count-limit=15000000 --internal-max-instruction-limit=15000000` (which works for SDXL UNet 1K) **does NOT lift the ceiling**. The check fires inside `HLOToTensorizer.runHlo2Tensorizer` (HLO verifier), before the tensorizer / instruction-count gating actually applies, so the user-supplied limit is silently ignored at this stage.

This blocks SDXL 2K/4K serving on trn2.3xlarge.

## Model
- **Name**: `stabilityai/stable-diffusion-xl-base-1.0` — VAE decoder (`AutoencoderKL.decoder` + `post_quant_conv`)
- **Type**: VAE decoder (Conv2d + ResnetBlock2D + UpDecoder2D + GroupNorm stack)
- **Size**: ~50M params (VAE decoder portion only)
- **Dtype**: bf16

## Simplified Reproducer

See `reproducer.py` in this directory (~80 lines).

The reproducer:
1. Loads only the VAE decoder + post_quant_conv from SDXL base 1.0 (bf16, fp16 variant).
2. Wraps them in a tiny `nn.Module` with `forward(latent) -> image`.
3. Traces with `torch_neuronx.trace()` at input shape `(1, 4, 256, 256)` (the latent for a 2048×2048 image).
4. Uses the standard "lift the ceiling" workaround flags `--tensorizer-options=--inst-count-limit=15000000 --internal-max-instruction-limit=15000000`.

No diffusers pipeline, no UNet, no text encoder, no scheduler, no actual SDXL ckpt loading beyond the VAE — just the VAE decoder graph, which is the failing site.

## Investigation Notes

### What we tried

| Configuration | Result |
|---|---|
| `--model-type=unet-inference --lnc=2 -O1` (no inst-count flag) | Same NCC_EVRF007, 7.69M instructions vs 5M limit |
| Add `--tensorizer-options=--inst-count-limit=15000000 --internal-max-instruction-limit=15000000` | **Same NCC_EVRF007** — flag is silently ignored at the HLO verifier stage |
| Lower res 1024² (latent `[1, 4, 128, 128]`) | Compiles fine — confirms the 2K shape specifically triggers the bug |

### Why we believe `--inst-count-limit` is silently ignored

The error fires inside the **HLO verifier pass** (`hilo/hlo_passes/NeuronHloVerifier.cc:1406`), before the tensorizer pass that consumes `--inst-count-limit`. The traceback shows:

```
neuronxcc/driver/jobs/HLOToTensorizer.py:205, runHlo2Tensorizer
```

So this is HLO→Tensorizer translation finishing and calling the verifier, which has its own internal limit (5,000,000) **separate from** the tensorizer-side `--inst-count-limit` flag.

If this hypothesis is correct, the customer-facing fix would be:
1. Either route `--inst-count-limit` (or a new `--hlo-instruction-limit`) through to the verifier
2. Or remove the verifier's hard 5M cap and rely on the downstream tensorizer pass for the actual limit

### Customer impact

This is the blocker for **SDXL 2K and 4K image generation** on trn2.3xlarge. SDXL 1024² compiles and runs fine with the same compile flags (we have a customer benchmark at 11.14s warm 1K with DP=2 + NKI flash attention). 2K/4K are customer-requested resolutions for OPPO POC.

A similar issue is also visible for **FLUX.1-dev 2048²** VAE decoder (NEFF 5.23M instructions, NCC_EVRF007) on the same SDK — same error class, separate ticket if useful.

### Confirmed bug still present in SDK 2.32

The previous attempt was on SDK 2.29 (`aws-neuronx-runtime-lib 2.x.47730.0`, prior DLAMI). This reproduction is on **SDK 2.32** (`neuronx-cc 2.25.3371.0+f524f7f8`, latest DLAMI Ubuntu 24.04 as of 2026-05-25). Bug behavior is identical.

## Asks

1. Is the 5M HLO verifier instruction limit configurable via a flag we are missing?
2. If not, is there a planned fix to either raise the limit or accept user override (similar to `--inst-count-limit` for downstream pass)?
3. Recommended workaround — TP-shard the VAE decoder, run on CPU, compile on a larger host RAM instance, etc.?
