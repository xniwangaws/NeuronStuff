## Title
SDXL VAE decoder NEFF compile fails with NCC_EVRF007 at 2048×2048; --inst-count-limit=15M flag does not lift the 5M HLO verifier ceiling

## Description (paste into ticket body)

### Summary

When tracing the SDXL VAE decoder for 2048×2048 output (input latent shape `[1, 4, 256, 256]`, bf16) on trn2.3xlarge, neuronx-cc aborts in the HLO verifier pass:

```
[NCC_EVRF007] Instructions generated by compiler 7,691,582 exceeds the typical
limit of 5,000,000.
```

The standard workaround `--tensorizer-options=--inst-count-limit=15000000 --internal-max-instruction-limit=15000000` (which works for SDXL UNet 1K) does NOT lift the ceiling at this stage. The check fires inside `HLOToTensorizer.runHlo2Tensorizer` (HLO verifier) before the tensorizer pass that consumes `--inst-count-limit`, so the user-supplied flag is silently ignored at this stage.

This blocks SDXL 2K/4K image generation on trn2.3xlarge for an active OPPO POC.

### Reproducer

Standalone, ~80 lines, no diffusers pipeline boilerplate. Loads only the SDXL VAE decoder + post_quant_conv (bf16) and traces with input `[1, 4, 256, 256]`.

```python
import os, time, torch, torch_neuronx
from diffusers import AutoencoderKL

vae = AutoencoderKL.from_pretrained(
    "stabilityai/stable-diffusion-xl-base-1.0",
    subfolder="vae", variant="fp16", torch_dtype=torch.bfloat16
).eval()

class VAEDecoderWrapper(torch.nn.Module):
    def __init__(self, post_quant_conv, decoder):
        super().__init__()
        self.post_quant_conv = post_quant_conv
        self.decoder = decoder
    def forward(self, latent):
        return self.decoder(self.post_quant_conv(latent))

wrapper = VAEDecoderWrapper(vae.post_quant_conv, vae.decoder).eval()
example = torch.randn(1, 4, 256, 256, dtype=torch.bfloat16)

torch_neuronx.trace(
    wrapper, example,
    compiler_workdir="/tmp/sdxl_2k_workdir",
    compiler_args=[
        "--model-type=unet-inference", "--lnc=2", "-O1",
        "--tensorizer-options=--inst-count-limit=15000000",
        "--internal-max-instruction-limit=15000000",
        "--verbose=DEBUG",
    ],
)
```

Failure occurs ~28s into the trace call, with exit code 70 (CompilerInvalidInputException).

### What we tried

| Configuration | Result |
|---|---|
| Default flags (`--model-type=unet-inference --lnc=2 -O1`) | NCC_EVRF007 (7.69M instructions vs 5M limit) |
| Add `--tensorizer-options=--inst-count-limit=15000000` + `--internal-max-instruction-limit=15000000` | **Same NCC_EVRF007** — flag silently ignored at HLO verifier stage |
| Lower res 1024² (latent `[1, 4, 128, 128]`) | Compiles fine. Confirms 2K shape specifically triggers the bug |

### Hypothesis

The error fires inside `hilo/hlo_passes/NeuronHloVerifier.cc:1406`, which appears to have its own internal limit (5,000,000) **separate from** the tensorizer-side `--inst-count-limit` flag. Stack trace shows:

```
neuronxcc/driver/jobs/HLOToTensorizer.py:205, runHlo2Tensorizer
```

Possible fixes:
1. Route `--inst-count-limit` (or a new `--hlo-instruction-limit`) through to the HLO verifier
2. Remove or raise the verifier's hard 5M cap and let downstream tensorizer enforce the user-supplied limit

### Customer impact

Active OPPO POC needs SDXL 2K and 4K image generation on trn2.3xlarge. SDXL 1024² runs fine (11.14s warm with DP=2 + NKI flash attention, 25% cheaper than H100 BF16). 2K/4K both blocked by this issue.

A related issue is visible for **FLUX.1-dev 2048²** VAE decoder (NEFF 5.23M instructions, same NCC_EVRF007). Likely same root cause — happy to file a separate ticket if useful.

### Asks

1. Is the 5M HLO verifier instruction limit configurable via a flag we are missing?
2. If not, planned fix or recommended workaround? (TP-shard VAE decoder? CPU fallback for VAE? Compile on larger host RAM and migrate NEFF? Tile-splitting the VAE decoder?)
