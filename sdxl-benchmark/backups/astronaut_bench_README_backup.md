**核心结论**:
- **H100 BF16 SDXL 在所有分辨率均是最快 + 单图成本最低**的选择(4K 3× VRAM 余量)
- L4 单价最低,但 1K 慢 5×,2K/4K 需要 `enable_vae_tiling` + `enable_model_cpu_offload` 才能跑;4K 单图 10 分钟 + 1-seed 抽样,**不适合生产**
- 4K 所有设备都正常出图(SDXL 不像 FLUX.2-klein 受 `max_area=4MP` 限制)

## 4. 硬件 / 软件配置

**H100 p5.4xlarge**:Deep Learning AMI / torch 2.9.1+cu128 / CUDA 12.9 / diffusers 0.38

**L4 g6.4xlarge**:DLAMI / torch 2.7.0+cu128 / diffusers 0.38。2K / 4K 需要 `enable_vae_tiling()` + `enable_model_cpu_offload()` 规避 VRAM OOM(SDXL UNet + VAE 在 4K 激活约 18 GB,超出 L4 24 GB 窗口)

## 5. Neuron trn2 SDXL 说明 —— SDK 2.29 compile 受限

尝试在 trn2.3xlarge (SDK 2.29, neuronx-cc 2.24.5133) 上 compile SDXL UNet,结果:
- **CLIP-L / CLIP-G / VAE 均 compile 成功**(2.24 GB NEFF,~3 min 编译)
- **UNet 1K compile 失败**:`neuronx-cc failed with 70`,原因为 `[NCC_EBVF030] Instructions generated by compiler 6,240,098 exceeds the typical limit of 5,000,000`
- **UNet 2K 尝试**:`23,812,228` 指令,失败
- 尝试 `--auto-cast=matmult --optlevel=1` + `UNET_CORES=0,1` 仍超限

**可行方案(未在本轮测试范围)**:
- (a) 升级到 SDK 2.30+(据称提高单 graph 指令上限)
- (b) 在 trn2.48xlarge 用 NxDI `NeuronUNet2DConditionConfig` 做 tensor parallelism 切分 UNet
- (c) 切分 UNet 为 down / mid / up 三段分别 trace(aws-neuron-samples 里有参考)

**当前状态**:trn2 SDXL 1K / 2K / 4K **N/A for SDK 2.29 on trn2.3xlarge**。跟 AWS Neuron 团队反馈后续迭代。

## 7. 结论

1. **H100 BF16 SDXL 是本场最佳性价比**:1K $0.00461 / 2K $0.01459 / 4K $0.11339,4K 单图用时 94s 可接受
2. **L4 BF16 稳定但慢**:1K 19.75s(5.14× 慢)、2K 95s、4K 619s 且仅 1-seed 抽样,**不推荐 4K 生产部署**
3. **SDXL 在 Neuron trn2 SDK 2.29 上受限**:UNet graph 超 5M 指令上限,**需要 trn2.48xlarge + TP 切分 or 等 SDK 升级**
4. **4K 可行性**:SDXL 在所有 GPU 设备上 4K 均能正常出图,无模型 spec 限制(对比 FLUX.2-klein `max_area=4MP` 导致 GRAY 噪声,SDXL 没这限制)
